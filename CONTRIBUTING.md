# Contributing to Chaos Node

Thank you for contributing to Chaos Node.

This project is being built with a strong focus on collaborative intelligence, governance-aware design, safe development, and clear review standards. Contributions are welcome, but they should align with the architecture, rules, and direction of the project.

## Before You Contribute

Please review the following files before submitting work:

* `README.md`
* `GOVERNANCE.md`
* `RULESETS.md`
* `RBAC.md`

Contributions should support the project's goals and should not undermine safety, governance, auditability, or clarity.

## Branching and Pull Requests

All contributions should be submitted to the `develop` branch for review.

Do not submit changes directly to protected or production-facing branches unless you have explicit authority to do so.

### Basic flow

1. Fork the repository or create a working branch.
2. Make your changes in a clearly named branch.
3. Keep changes focused and easy to review.
4. Submit your pull request into `develop`.
5. Wait for review before merge.

## Contribution Standards

When contributing:

* keep changes clear and intentional
* document important assumptions
* avoid hidden behavior changes
* do not bypass governance or safety logic
* follow existing project structure where possible
* write with maintainability in mind

## High-Review Change Areas

The following areas may require closer review:

* governance changes
* ruleset changes
* RBAC changes
* tool execution logic
* deployment logic
* training ingestion logic
* memory and conflict handling logic

## Proposing Big Changes

If you want to propose a major feature, architectural shift, or governance change, open an issue first so the direction can be reviewed before significant work is submitted.

## Final Note

Chaos Node is meant to be ambitious, structured, and responsibly built.

Good contributions improve capability without weakening control, clarity, or trust.

Submit to `develop`, keep your work reviewable, and help us build this correctly.
